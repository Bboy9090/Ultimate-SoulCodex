# Soul Codex Output Schema v1 (Canonical Contract)

This repository now exposes a single canonical, machine-readable profile output contract:

- **Endpoint**: `GET /api/profiles/:id/soul-codex-v1?tone_mode=clean|deep|raw`
- **Version**: `soul_codex_v1`

This contract is designed to be **stable, shareable, and testable**. It intentionally separates:

- **Core system** (fixed data)
- **Identity summary** (tight first interpretation)
- **Sections** (structured interpretation blocks)
- **Daily guidance** (strict actionable object)

## Source of truth

- **Schema (zod + TS types)**: `packages/core/soulcodex-v1/schema.ts`
- **Deterministic generator (no AI, internal engine-backed)**: `packages/core/soulcodex-v1/generate.ts`
- **API route**: `apps/api/routes.canonical.ts` (`/api/profiles/:id/soul-codex-v1`)

## Life Event Decoder (deterministic)

The app also exposes a deterministic Life Event Decoder for turning a user-described situation into a structured, actionable response. This is intended for **clarity and next-step guidance**, not “prediction.”

- **Endpoint**: `POST /api/events/decode`
- **API route**: `apps/api/routes.canonical.ts`
- **Core implementation**: `packages/core/events/decode.ts` (`decodeLifeEvent`)
- **Types**: `packages/core/events/types.ts`

### Request body

- **Preferred**
  - `text`: string (required)
  - `category_hint`: optional string (one of the `LifeEventCategory` union)
  - `phase`: optional string
  - `driver`: optional string

- **Back-compat (legacy)**
  - `event`: string (treated as `text`)
  - `phase`: string (optional)
  - `driver`: string (optional)

### Response body

Returns `LifeEventDecode`:

- `category`: string
- `intensity`: `"low" | "medium" | "high"`
- `meaning`: string
- `next_steps`: string[]
- `avoid`: string[]
- `decision_rule`: string
- `follow_up_questions`: string[]

Back-compat keys are also included for older clients:

- `whyNow`: alias of `meaning`
- `lesson`: alias of `decision_rule`
- `nextSteps`: alias of `next_steps`

## Internal engine (Matrix v1)

The public contract (`SoulCodexOutputV1`) is generated by an internal deterministic engine:

- **Engine entrypoint**: `packages/core/soulcodex-v1/engine/index.ts`
- **Content libraries** (JSON, imported deterministically — no runtime fs path tricks):
  - `packages/core/soulcodex-v1/matrix/trait-mapping-matrix.json`
  - `packages/core/soulcodex-v1/rules/*.json`
  - `packages/core/soulcodex-v1/content/statement-library/*.json`
- **Loader + validation**: `packages/core/soulcodex-v1/engine/loaders.ts` (Zod-validated)

Pipeline:

1. **Trait mapping**: input signals → `TraitSignal[]` (`engine/traitMapper.ts`)
2. **Contradictions**: remove blocked traits deterministically (`engine/contradictions.ts`)
3. **Statement selection**: tone-aware, confidence-thresholded, banned-language filtered (`engine/statementSelector.ts`)
4. **Daily guidance**: deterministic daily object from trait hints (`engine/dailyGuidance.ts`)
5. **Adapter**: map engine output into `identity_summary`, `sections`, `daily_guidance` (`generate.ts`)

Mirror (behavioral) layer:

- Optional user answers (`profile.mirror` or `profile.mirrorAnswers`) are analyzed deterministically (`engine/mirror.ts`) and converted into extra `TraitSignal[]` that sharpen statement selection + daily guidance without changing the public schema.

## Payload shape (v1)

The server returns a JSON object matching `SoulCodexOutputV1`:

- `profile_id`: string
- `version`: `"soul_codex_v1"`
- `generated_at`: ISO datetime
- `confidence`:
  - `astrology`: `verified|partial|unverified`
  - `human_design`: `verified|partial|unverified`
  - `numerology`: `verified|partial|unverified`
  - `overall`: `high|medium|low`
- `input_summary`:
  - `birth_date`: `YYYY-MM-DD` (or `"unknown"` if missing)
  - `birth_time`: optional `HH:MM`
  - `birth_location`: optional string
- `core_system`: `{ astrology, human_design, numerology }` (fixed-data layer)
- `identity_summary`: `{ codename, one_line, system_translation }`
- `sections`: `{ how_you_work, decision_code, relational_pattern, failure_conditions, optimal_conditions, distortion_mode, power_mode, mission_arc }`
- `daily_guidance`: `{ title, focus, do[], dont[], watch_outs[], decision_advice }`
- `tone_mode`: `clean|deep|raw` (mode hint; v1 generator is deterministic and mode-neutral for now)

## Extending v1 (next build pieces)

This schema is the **contract**. Improvements should not break the shape.

Recommended next steps (incremental, reviewable):

1. Expand matrix + statement library coverage (more signals, more statements per section).
2. Add “insight trace” metadata in a debug-safe way (without breaking the public contract).
3. Tighten weighting rules (currently loaded/validated; selection is overlap+confidence based).
4. Add more fixture tests for edge profiles (no time, no location, missing systems).

## Notes

- The generator is currently **deterministic** and leverages existing stored fields like `profile.astrologyData`, `profile.humanDesignData`, `profile.numerologyData`, and `profile.synthesis` when present.
- The contract is enforced server-side via zod parse (`soulCodexOutputV1Schema.parse(...)`).
